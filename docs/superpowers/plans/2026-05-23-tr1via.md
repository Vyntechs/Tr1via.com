# TR1VIA Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan phase-by-phase. Steps use checkbox (`- [ ]`) syntax.

**Goal:** Build the working TR1VIA web app — Jeopardy-style live multiplayer trivia for one venue at a time — matching the Claude Design package pixel-perfect, against `tr1via-plan.md` rules.

**Architecture:** Single Next.js 16 (App Router) app. Three surfaces share the codebase: player phones, the venue TV (mirrored from host laptop), and the host (laptop + private phone). The database (Supabase Postgres) is the single source of truth; every screen subscribes to it via Supabase Realtime. The host presses Reveal → one timestamped DB event → TV + every phone update within ~150ms.

**Tech Stack:**
- **Framework:** Next.js 16 + React 19 + TypeScript (App Router, Server Components, Server Actions)
- **Styling:** Tailwind CSS v4 (utility) + CSS custom properties (theme tokens, runtime-swapped)
- **Fonts:** `next/font` (Geist + Geist Mono), Google Fonts (Bricolage Grotesque)
- **DB + Realtime + Auth + Storage:** Supabase
- **AI:** Claude API (`claude-sonnet-4-6`) for question generation
- **Photos:** Pexels API (auto-attach) + Supabase Storage (manual upload)
- **Test:** Vitest (unit), React Testing Library (component), Playwright (E2E + multi-device sync)
- **Deploy:** Vercel (preview per PR, prod = tr1via.com)

## Key decisions (locked, with rationale)

| Decision | Choice | Why |
|---|---|---|
| Database | **Supabase Postgres** (not Neon as plan.md said) | Supabase ships Postgres + Realtime + Auth + Storage in one managed service. Neon alone has no real-time fan-out — we'd need a separate Pusher/Ably layer. Supabase is a single dependency that delivers the spec's "one press, three surfaces" requirement out of the box. Brandon: if you want Neon for cost/data-residency reasons, flag and we migrate the data layer (the app code is portable). |
| Real-time transport | **Supabase Realtime** (Postgres Changes + Broadcast channels) | Free tier covers our scale (2M messages/mo, ~30 concurrent devices per room). Broadcast for ephemeral game events (reveal, tick), Postgres Changes for durable state (roster, scores). |
| Auth — host | **Supabase Auth · magic link** (email only, no password) | Simplest path that doesn't require Brandon to manually provision each host. Designer said this is Claude Code's call. |
| Auth — player | **Anonymous device session** — UUID in httpOnly cookie + localStorage | Players don't have accounts. Cookie persists across both games of a night. Cleared by browser when they want. |
| AI | **Anthropic Claude API · claude-sonnet-4-6** | Sonnet for cost+latency. Use the `claude-api` skill for prompt-caching + structured output. |
| Photos | **Pexels API** (free, commercial-license stock) + **Supabase Storage** for uploads | Designer named Pexels as the example. Free 200 req/hr, 20K req/mo on free tier. |
| Styling | **Tailwind v4 utility + CSS custom properties for theme** | Theme tokens (14 palettes × 6 colors) live as CSS vars on a `[data-theme="house"]` root selector. Components mix Tailwind utilities for layout + inline styles referencing `var(--accent)` for theme colors. Lets us runtime-swap themes without re-render. |
| Repo layout | **Single Next.js app at repo root** | Player/TV/host all share the design system and DB schema — one app, route groups for surfaces. Splitting into multiple apps would force a shared package and worsen deploy. |
| Tests | **Vitest + RTL + Playwright multi-context** | Playwright can drive 3+ browser contexts in one test → the only practical way to verify "host presses → TV + phones in sync." |

These are commits, not proposals. Plan executes against them.

import { TR1VIA_THEMES, type ThemeKey } from "./tokens";
import { TR1VIA_CATEGORIES } from "./categories";

export function resolveTheme(themeKey: ThemeKey = "house") {
  const t = TR1VIA_THEMES[themeKey] ?? TR1VIA_THEMES.house;
  const dark = t.mode === "dark";
  return {
    ...t,
    surface:  dark ? "rgba(255,255,255,.04)"  : "rgba(27,19,12,.04)",
    surfaceH: dark ? "rgba(255,255,255,.08)"  : "rgba(27,19,12,.08)",
    inkMid:   dark ? "rgba(244,230,196,.62)" : "rgba(27,19,12,.62)",
    inkMute:  dark ? "rgba(244,230,196,.36)" : "rgba(27,19,12,.36)",
    line:     dark ? "rgba(244,230,196,.14)" : "rgba(27,19,12,.13)",
    lineSoft: dark ? "rgba(244,230,196,.07)" : "rgba(27,19,12,.06)",
    dark,
    categories: TR1VIA_CATEGORIES,
  };
}
export type ResolvedTheme = ReturnType<typeof resolveTheme>;

/**
 * Award per the rules:
 *   - correct → face value
 *   - correct AND msToLock < 5000 → +10% (floor)
 *   - wrong / no answer → 0
 */
export function awardPoints(opts: { pointValue: number; correct: boolean; msToLock: number | null }): number {
  if (!opts.correct) return 0;
  const bonus = opts.msToLock !== null && opts.msToLock < 5000;
  return bonus ? Math.floor(opts.pointValue * 1.1) : opts.pointValue;
}

### Task 4.3: Timer

`lib/game/timer.ts`:

```ts
/**
 * Seconds remaining, clamped to [0, durationS]. Uses serverNow if provided
 * to compensate for client clock skew (server sends `now` in DB events).
 */
export function secondsRemaining(opts: { revealedAtMs: number; durationS: number; nowMs?: number }): number {
  const now = opts.nowMs ?? Date.now();
  const elapsedS = (now - opts.revealedAtMs) / 1000;
  return Math.max(0, Math.min(opts.durationS, opts.durationS - elapsedS));
}

/** Fractional 0..1 progress for arc rendering. */
export function timerFraction(opts: { revealedAtMs: number; durationS: number; nowMs?: number }): number {
  return secondsRemaining(opts) / opts.durationS;
}
```

Tests: revealed-now → 20s; revealed 5s ago → 15s; revealed 25s ago → 0s; never < 0.

### Task 6.3: Reveal flow

Critical task — TDD with multi-context Playwright at the end.

- [ ] `POST /api/games/:id/reveal` `{ questionId }`:
  1. Begin transaction
  2. Insert into `reveals` (event='reveal', occurred_at=now())
  3. Update `questions.played_at = now()`
  4. Commit
  5. Broadcast on Supabase Realtime channel `room:{roomCode}` event `reveal` with `{ questionId, revealedAt: ISO }`

- [ ] Subscribers (TV, every phone): on `reveal` event, fetch the question + start local timer based on `revealedAt`.

- [ ] `POST /api/games/:id/undo` (within 2s of reveal, host-only): inserts event='undo', deletes any answers cast in the window, clears `played_at`. Broadcast `undo`.

- [ ] `POST /api/games/:id/end-early` `{ questionId }`: shortcuts to resolve (Phase 6.5).

### Task 6.4: Submit answer

- [ ] `POST /api/answers` `{ questionId, slotChosen, scramble }` (player-only):
  - Verify question state is "live" (played_at is set, not yet finished_at).
  - Verify scramble matches what server computes for `scrambleFor(questionId, playerId)` (anti-tamper).
  - Insert into `answers` with `chosen_index = scramble[slotChosen-1]`, `ms_to_lock = now - revealed_at`.
  - Return acknowledgement (NOT correctness; that's revealed at T+20).

### Task 6.5: Resolve question (at T+20 or end-early)

- [ ] A server-side scheduler (Vercel Cron isn't real-time; instead use a Supabase Edge Function triggered by the broadcast). Simplest: when the first phone's timer hits 0, it pings `POST /api/questions/:id/resolve` (idempotent — server checks if already resolved).
- [ ] In `resolve`:
  - Update `questions.finished_at = now()`
  - For every answer: compute `is_correct` (compare `chosen_index` to `correct_index`), compute `awarded_points` via `awardPoints()`
  - Insert into `reveals` (event='resolve')
  - Broadcast `resolve` with `{ questionId, correctIndex, awards: [{playerId, awarded, isCorrect}] }`
- [ ] Make this race-safe: use `select for update` on questions row.
